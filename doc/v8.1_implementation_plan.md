# v8.1 Implementation Plan: SetState Subtree Count O(N^2) ŌåÆ O(N)

## Problem

`SetStateScopeDetector.checkElement()` (`setstate_scope_detector.dart:143-149`) runs a nested recursive walk (`countSubtree`) for every qualifying `StatefulElement`. For S stateful widgets and N total elements, this is O(S├ŚN) ŌĆö effectively O(N^2). Since scans run on the UI thread every ~1s, the watchdog can create its own jank on large trees.

### Current O(N^2) code (`setstate_scope_detector.dart:142-156`)

```dart
// Inside checkElement, for each qualifying StatefulElement:
int subtreeSize = 0;
void countSubtree(Element child) {
  subtreeSize++;
  child.visitChildren(countSubtree);
}
element.visitChildren(countSubtree);

if (subtreeSize > _maxSubtreeSize) {
  _maxSubtreeSize = subtreeSize;
  _widestStatefulWidget = name;
  _widestElement = element;
}
```

This nested walk inside `checkElement` defeats v7.9's unified O(N) tree walk (`watchdog_controller.dart:784-792`).

### Unified walk structure (context)

The controller's unified walk (`watchdog_controller.dart:785-792`) calls detectors in this order:

```dart
void visitor(Element element) {
  for (final d in unified) {
    d.checkElement(element);     // pre-order
  }
  element.visitChildren(visitor); // recurse into children
  for (final d in unified) {
    d.afterElement(element);     // post-order
  }
}
```

`afterElement` is called AFTER all children have been visited ŌĆö confirmed post-order semantics. Two existing detectors already use this pattern:
- `NestedScrollDetector.afterElement` ŌĆö pops from `_scrollAxisStack` (line 72-74)
- `ShallowRebuildRiskDetector.afterElement` ŌĆö decrements `_depth` (line 114-116)

## Solution: Stack-based post-order subtree size accumulation

Replace the nested `countSubtree` walk with a `List<int> _subtreeSizeStack` that accumulates subtree sizes during the unified walk's existing `checkElement`/`afterElement` lifecycle.

### Algorithm

1. **`checkElement`** (every element): Push `0` onto `_subtreeSizeStack`
2. **`afterElement`** (every element): Pop from stack. Add `poppedValue + 1` to parent's stack top (the `+1` counts the current element). If element is a qualifying `StatefulElement`, check if it's the new max.
3. **`finalizeScan`**: Read `_maxSubtreeSize` / `_widestStatefulWidget` / `_widestElement` directly ŌĆö no lookup needed.

### Mathematical equivalence proof

For any element E with descendants DŌéü, DŌéé, ..., DŌé¢:

**Old code:** Recursively count all descendants via `countSubtree` = total elements below E.

**New code trace:**
```
checkElement(E): push 0
  For each child CßĄó of E:
    checkElement(CßĄó): push 0
      ... (CßĄó's subtree fully processed)
    afterElement(CßĄó): pop sßĄó, E.top += sßĄó + 1
afterElement(E): pop value = ╬Ż(sßĄó + 1) = ╬Ż(sßĄó) + childCount = total descendants
```

Both compute identical values. Verified with concrete examples:

**Example tree:** `LargePageWidget ŌåÆ Column ŌåÆ 30├ŚSizedBox`
- Old: `countSubtree` visits Column(1) + 30 SizedBoxes(31) = 31
- New: Each SizedBox contributes 0+1=1 to Column. Column accumulates 30. Column contributes 30+1=31 to LargePageWidget. Pop = 31. Ō£ō

**Example chain:** `A ŌåÆ B ŌåÆ C ŌåÆ D`
- Old: A counts B(1)+C(2)+D(3) = 3
- New: D pops 0, C.top += 1. C pops 1, B.top += 2. B pops 2, A.top += 3. A pops 3. Ō£ō

#### Change 5: Add `afterElement` override (new method, after `checkElement`)

```dart
@override
void afterElement(Element element) {
  final subtreeSize = _subtreeSizeStack.removeLast();
  if (_subtreeSizeStack.isNotEmpty) {
    _subtreeSizeStack.last += subtreeSize + 1;
  }
  // Track max inline ŌĆö same filters as checkElement (lines 119-120):
  // skip private-named and framework-owned StatefulWidgets so that
  // Scaffold, Navigator, Overlay, etc. never become the "widest" candidate.
  if (element is StatefulElement && subtreeSize > _maxSubtreeSize) {
    final name = element.widget.runtimeType.toString();
    if (!name.startsWith('_') && !isFrameworkWidget(element.widget)) {
      _maxSubtreeSize = subtreeSize;
      _widestStatefulWidget = name;
      _widestElement = element;
    }
  }
}
```

#### Change 6: Add abort-safety guard to `finalizeScan` (line 162)

Both `BaseDetector.scanTree()` (`base_detector.dart:137-140`) and the controller unified walk (`watchdog_controller.dart:795-797`) swallow traversal exceptions with `try {} catch (_) {}` and still call `finalizeScan()`. If the walk aborts mid-tree, `afterElement()` won't run for all pushed elements ŌĆö the stack will be non-empty. Without a guard, `finalizeScan` would read `_maxSubtreeSize` from a partial traversal, potentially emitting issues based on incomplete data.

```dart
// Current (line 162):
@override
void finalizeScan() {
  // Swap snapshots
  _childSnapshots = _newSnapshots;

  if (_totalElements == 0 || _maxSubtreeSize < minSubtreeSize) return;
  // ...

// New:
@override
void finalizeScan() {
  // Abort safety: if the walk was interrupted (exception swallowed by
  // try/catch in unified walk or scanTree), the stack won't be fully
  // drained. Discard the partial scan entirely ŌĆö partial subtree sizes
  // are unreliable, and partial _newSnapshots would lose the complete
  // rebuild baseline for all unvisited StatefulElements.
  if (_subtreeSizeStack.isNotEmpty) {
    _subtreeSizeStack.clear();
    // Do NOT swap _childSnapshots = _newSnapshots here.
    // Keep the last fully-completed snapshot so the next scan's
    // rebuild detection has a complete baseline to compare against.
    // Partial _newSnapshots only covers elements visited before the
    // exception ŌĆö promoting it would create gaps where rebuild
    // evidence is silently lost.
    return;
  }

  // Swap snapshots
  _childSnapshots = _newSnapshots;

  if (_totalElements == 0 || _maxSubtreeSize < minSubtreeSize) return;
  // ...
```

On abort, the snapshot swap is **not** performed. The last fully-completed snapshot is preserved as the rebuild baseline. This prevents partial scans from creating evidence gaps that suppress real wide-setState detections.

## Memory analysis

- **Stack depth:** Grows to max tree depth (not node count). Flutter widget trees rarely exceed depth ~100. `List<int>` at depth 100 = ~800 bytes.
- **Stack lifecycle:** Populated and fully drained within a single scan. No per-scan element references survive in the stack.
- **vs. Map<Element, int> alternative:** The spec explicitly rejects a per-element map because the controller's navigation early-return path (`watchdog_controller.dart:608-624`) returns BEFORE `prepareScan` runs. A map would retain the previous page's entire element graph until the next successful scan. Stack-only + scalar fields avoids this entirely.

## Retention fix: `clearSnapshots()` expansion

**The problem:** `_widestElement` (Element?) and `_widestStatefulWidget` (String?) survive navigation aborts:

```
_scanTree called ŌåÆ _findVisiblePageContext returns null (transition) ŌåÆ
  early return at line 608-624 ŌåÆ
  calls clearSnapshots() (line 622) ŌåÆ
  BUT clearSnapshots() only clears _childSnapshots and _rebuildEvidence ŌåÆ
  _widestElement still holds reference to previous page's Element ŌåÆ
  prepareScan (which nulls it at line 108) never runs during aborted scans
```

**The fix:** Null both fields in `clearSnapshots()`. This ensures no element reference survives across navigation-abort cycles.

## Risk assessment

**Risk: Low.**
- Pure algorithmic refactor ŌĆö same detection logic, same outputs
- No new fields consumed by external code
- Stack push/pop pairing holds under normal walk completion; abort-safety guard (`_subtreeSizeStack.isNotEmpty` check in `finalizeScan`) handles exception-aborted walks by discarding partial results
- `clearSnapshots()` expansion is strictly an improvement (closes retention gap); verified via controller-level navigation-abort test
- Benchmark suite updated to prove O(N) scaling and prevent regression
